# 企画書作成

## 概要

WebSearch / WebFetch で市場データ・トレンド・競合分析を収集し、proposals/inbox/{name}/ に企画書を作成する。

## 手順

### Step 1: データ収集（WebSearch / WebFetch）

以下のデータを WebSearch / WebFetch で収集する:

**市場規模:**
- TAM（Total Addressable Market）: 市場全体の規模
- SAM（Serviceable Addressable Market）: 自社がアプローチ可能な市場
- SOM（Serviceable Obtainable Market）: 現実的に獲得可能な市場
- CAGR（年平均成長率）

**検索クエリ例:**
- "{テーマ} market size 2025"
- "{テーマ} CAGR forecast"
- "{テーマ} industry report"

**トレンド情報:**
- テクノロジートレンド（新技術、AI 活用等）
- 消費者トレンド（行動変化、ニーズの変化等）
- 直近 6 ヶ月のニュース・動向

**検索クエリ例:**
- "{テーマ} trends 2025"
- "{テーマ} technology innovation"
- "{テーマ} consumer behavior"

**競合分析:**
- 主要競合 3 社以上
- 各競合の機能・価格・ユーザー数
- 競合の強み・弱み
- Gap 分析（ブルーオーシャン機会）

**検索クエリ例:**
- "{テーマ} competitors"
- "{テーマ} alternatives comparison"
- "{競合名} features pricing"

**重要:**
- 英語と日本語の両方で検索する
- 公式サイトの情報を WebFetch で取得する
- ソースの日付に注意（古いデータは明記する）
- 複数ソースでクロスチェックする（重要データは 2 つ以上のソースで裏付け）
- 一次ソース（公式発表、政府統計、学術論文）を優先する

### Step 2: ディレクトリ作成

```bash
mkdir -p proposals/inbox/{proposal-slug}/
```

プロポーザルスラッグは英語の kebab-case で命名する（例: `ai-code-review-tool`, `saas-analytics`）

### Step 3: 企画書本体（proposal.md）

以下のテンプレートに従い企画書を作成する。

```markdown
# {プロダクト名}

## ステータス
draft / under-review / approved / rejected

## エレベーターピッチ
{1-2文でプロダクトを説明。誰の・何の課題を・どう解決するか}

## 課題
### 現状の問題
{ターゲットユーザーが抱えている具体的な課題}

### なぜ今か
{このタイミングで解決すべき理由。トレンドとの関連}

## ターゲットユーザー
| セグメント | ペルソナ | ペインポイント |
|-----------|---------|-------------|
| {セグメント1} | {具体的なペルソナ} | {課題} |
| {セグメント2} | {具体的なペルソナ} | {課題} |

## コンセプト
### ソリューション概要
{プロダクトの全体像}

### 差別化ポイント
{競合分析から導出された明確な差別化。なぜ既存プロダクトではダメか}

## MVP（Minimum Viable Product）
### 機能一覧
| # | 機能 | 説明 | 優先度 |
|---|------|------|--------|
| 1 | {機能名} | {説明} | Must |
| 2 | {機能名} | {説明} | Must |
| 3 | {機能名} | {説明} | Must |
| 4 | {機能名} | {説明} | Should |
| 5 | {機能名} | {説明} | Could |

### MVP で検証すること
{MVP で何を検証するか。成功基準}

## 市場規模
| 指標 | 値 | ソース |
|------|-----|--------|
| TAM | ${N}B | [ソース](URL) |
| SAM | ${N}B | [ソース](URL) |
| SOM | ${N}M | 推定 |

## ビジネスモデル
### 収益化
{フリーミアム / サブスクリプション / 従量課金 etc.}

### 料金プラン（案）
| プラン | 月額 | 対象 | 主要機能 |
|--------|------|------|---------|
| Free | $0 | {対象} | {機能} |
| Pro | ${N} | {対象} | {機能} |

## リスクと対策
| リスク | 影響度 | 対策 |
|--------|--------|------|
| {リスク1} | 高/中/低 | {対策} |
| {リスク2} | 高/中/低 | {対策} |

## 技術概要
### 技術スタック（推奨）
{プロダクトに適した技術スタックの提案}

### アーキテクチャ概要
{高レベルのアーキテクチャ}
```

### Step 4: PR 作成

```bash
gh pr create --repo machina-gg/trillion-game-lab \
  --title "feat: {プロダクト名}の企画書" \
  --body "## Summary
- proposal.md: 企画書本体（市場調査・データ収集を含む）

## コンセプト
{エレベーターピッチ}

Closes #{issue-number}"
```

## 企画書のポイント

- **エレベーターピッチ**: 1-2 文で説明できるか。「誰の・何の課題を・どう解決するか」
- **差別化**: 競合分析から導出された明確な差別化。「なぜ既存プロダクトではダメか」を説明できること
- **MVP**: 最小限の機能で検証可能な範囲に絞る。3-5 機能が目安
- **データ駆動**: 全ての主張にソース URL と取得日の裏付けがあること

## 品質基準

- [ ] エレベーターピッチが 1-2 文で明確
- [ ] 差別化が競合分析に基づいている
- [ ] MVP が 3-5 機能に絞られている
- [ ] 全データにソース URL と取得日が記載されている
- [ ] ビジネスモデルが記載されている
- [ ] リスクと対策が検討されている
