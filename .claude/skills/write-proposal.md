# 企画書作成

## 概要

research/{theme}/ のリサーチ結果を元に、proposals/{name}/ に企画書を作成する。

## 手順

### Step 1: リサーチの読み込み

research/{theme}/ の全ドキュメントを読む:
- market-analysis.md: 市場規模・成長率
- trend-report.md: トレンド情報
- competitor-analysis.md: 競合分析
- sources.md: ソース一覧

不足しているデータがあれば、Researcher に追加調査を依頼する（`/inter-agent-protocol` の追加データ依頼テンプレートを使用）。

### Step 2: ディレクトリ作成

```bash
mkdir -p proposals/{proposal-slug}/
```

プロポーザルスラッグは英語の kebab-case で命名する（例: `ai-code-review-tool`, `saas-analytics`）

### Step 3: 企画書本体（proposal.md）

以下のテンプレートに従い企画書を作成する。

```markdown
# {プロダクト名}

## ステータス
draft / under-review / approved / rejected

## エレベーターピッチ
{1-2文でプロダクトを説明。誰の・何の課題を・どう解決するか}

## 課題
### 現状の問題
{ターゲットユーザーが抱えている具体的な課題}

### なぜ今か
{このタイミングで解決すべき理由。トレンドとの関連}

## ターゲットユーザー
| セグメント | ペルソナ | ペインポイント |
|-----------|---------|-------------|
| {セグメント1} | {具体的なペルソナ} | {課題} |
| {セグメント2} | {具体的なペルソナ} | {課題} |

## コンセプト
### ソリューション概要
{プロダクトの全体像}

### 差別化ポイント
{競合分析から導出された明確な差別化。なぜ既存プロダクトではダメか}

## MVP（Minimum Viable Product）
### 機能一覧
| # | 機能 | 説明 | 優先度 |
|---|------|------|--------|
| 1 | {機能名} | {説明} | Must |
| 2 | {機能名} | {説明} | Must |
| 3 | {機能名} | {説明} | Must |
| 4 | {機能名} | {説明} | Should |
| 5 | {機能名} | {説明} | Could |

### MVP で検証すること
{MVP で何を検証するか。成功基準}

## 市場規模
| 指標 | 値 | ソース |
|------|-----|--------|
| TAM | ${N}B | [ソース](URL) |
| SAM | ${N}B | [ソース](URL) |
| SOM | ${N}M | 推定 |

## ビジネスモデル
### 収益化
{フリーミアム / サブスクリプション / 従量課金 etc.}

### 料金プラン（案）
| プラン | 月額 | 対象 | 主要機能 |
|--------|------|------|---------|
| Free | $0 | {対象} | {機能} |
| Pro | ${N} | {対象} | {機能} |

## リスクと対策
| リスク | 影響度 | 対策 |
|--------|--------|------|
| {リスク1} | 高/中/低 | {対策} |
| {リスク2} | 高/中/低 | {対策} |

## 技術概要
### 技術スタック（推奨）
{プロダクトに適した技術スタックの提案}

### アーキテクチャ概要
{高レベルのアーキテクチャ}
```

### Step 4: リサーチ参照（research-ref.md）

リサーチへの参照リンクを記載する。

```markdown
# リサーチ参照

## 元リサーチ
- テーマ: {テーマ名}
- ディレクトリ: `research/{theme}/`

## 参照ドキュメント
| ドキュメント | 参照箇所 |
|------------|---------|
| market-analysis.md | 市場規模セクション |
| trend-report.md | 「なぜ今か」セクション |
| competitor-analysis.md | 差別化ポイント、機能一覧 |
| sources.md | 全データのソース |

## 追加データ
{企画書作成中に追加で調査したデータがあれば記載}
```

### Step 5: PR 作成

```bash
gh pr create --repo machina-gg/trillion-game-lab \
  --title "feat: {プロダクト名}の企画書" \
  --body "## Summary
- proposal.md: 企画書本体
- research-ref.md: リサーチへの参照

## コンセプト
{エレベーターピッチ}

Closes #{issue-number}"
```

## 企画書のポイント

- **エレベーターピッチ**: 1-2文で説明できるか。「誰の・何の課題を・どう解決するか」
- **差別化**: 競合分析から導出された明確な差別化。「なぜ既存プロダクトではダメか」を説明できること
- **MVP**: 最小限の機能で検証可能な範囲に絞る。3-5機能が目安
- **データ駆動**: 全ての主張にリサーチデータの裏付けがあること

## 品質基準

- [ ] エレベーターピッチが1-2文で明確
- [ ] 差別化が競合分析に基づいている
- [ ] MVPが3-5機能に絞られている
- [ ] リサーチデータの参照がある
- [ ] ビジネスモデルが記載されている
- [ ] リスクと対策が検討されている
