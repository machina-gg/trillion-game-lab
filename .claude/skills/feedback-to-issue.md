# フィードバック → Issue 変換

## 概要

ユーザーからフィードバック（改善提案、テーマ提案、プロセス改善）を受けた際に、
即座に GitHub Issue に変換してサイクルに乗せる。

**原則: フィードバックを受けたら、その場で Issue にする。「後で」は禁止。**

## 実施タイミング

- ユーザーからフィードバックを受けた時（会話中に即実行）
- 振り返り（`/retrospective`）で改善点が出た時（retrospective 内から呼び出し可）

## 手順（3ステップ）

### Step 1: フィードバックの分類

ユーザーの発言を以下に分類する:

| 分類 | 判定基準 | Issue プレフィックス | ラベル |
|------|---------|--------------------|----|
| **テーマ提案** | 「〜を調べて」「〜のリサーチ」 | `feat:` | `research` |
| **企画要望** | 「〜の企画を」「〜を企画して」 | `feat:` | `proposal` |
| **改善提案** | 「〜を見直したい」「〜の仕組みを変えたい」 | `feat:` or `refactor:` | `improvement` |
| **プロセス改善** | 「フローが〜」「仕組みが〜」 | `chore:` | `improvement` |

### Step 2: 対象リポジトリの判定

| フィードバックの対象 | リポジトリ | `--repo` 指定 |
|-------------------|----------|-------------|
| リサーチ・企画の内容 | 企画部リポ | `machina-gg/trillion-game-lab` |
| 開発プロセス・ルール・スキル | 企画部リポ | `machina-gg/trillion-game-lab` |
| プロダクト開発に関すること | 開発部リポ | `machina-gg/trillion-game` |
| 不明 | ユーザーに確認 | -- |

### Step 3: Issue 作成

```bash
gh issue create \
  --repo machina-gg/trillion-game-lab \
  --title "{prefix} {フィードバックを1行に要約}" \
  --label "{label}" \
  --body "$(cat <<'EOF'
## 背景
ユーザーからのフィードバック（{日付}）

## 原文
> {ユーザーの発言をそのまま引用}

## 要件
- [ ] {フィードバックから導出した具体的な要件 1}
- [ ] {要件 2}
- [ ] {要件 3}

## 備考
{PM が補足する文脈}
EOF
)"
```

**作成後、Issue URL をユーザーに報告する。**

## 複数フィードバックの一括処理

ユーザーが複数のフィードバックを一度に出した場合:

1. 各フィードバックを個別に分類する
2. **1フィードバック = 1 Issue** で作成する（まとめない）
3. 作成した Issue の一覧をユーザーに報告する

## チェックリスト

- [ ] フィードバックを分類した（テーマ提案/企画要望/改善提案/プロセス改善）
- [ ] 対象リポジトリを判定した
- [ ] `gh issue create` で Issue を作成した
- [ ] Issue URL をユーザーに報告した
