# エージェント間 直接通信プロトコル

## 概要

技術的なやり取りは PM を経由せず、エージェント同士が直接通信する。
PM は「薄いルーター」であり、判断のボトルネックにしない。

## 直接通信が許可されるケース

| 送信者 | 受信者 | ケース |
|--------|--------|--------|
| Researcher | Planner | リサーチデータの共有、分析結果の引き渡し |
| Planner | Researcher | 追加データの依頼、不足情報のリクエスト |
| Planner | Editor | 企画書の事前相談、方向性の確認 |
| Editor | Planner | レビューフィードバック、修正指示 |
| Editor | Researcher | リサーチデータのファクトチェック依頼 |
| Researcher | Editor | データ訂正・補足の報告 |

## PM にエスカレーションが必要なケース

以下は直接通信ではなく、PM に報告する:

- **スコープ変更**: ユーザー要件に影響する変更
- **テーマ変更**: リサーチ対象の大幅な方向転換
- **ブロッカー**: 自力で解決できない問題
- **品質懸念**: リサーチデータの信頼性に重大な問題
- **権限不足**: 必要な操作ができない
- **開発部連携**: trillion-game リポへの Issue 作成等

## 通信手段

### 1. PR コメント（推奨）

最も透明性が高い方法。レビューの文脈でやり取りする。

```bash
# PR にコメントを追加
gh pr comment {PR番号} --body "Planner へ: この市場規模データは出典が弱いので、追加ソースを入れてください"
```

### 2. Task ツール直接メッセージ

同一チーム内のエージェント同士で SendMessage を使用する。

## リサーチデータ共有テンプレート（Researcher → Planner）

```
テーマ: {リサーチテーマ}
成果物: research/{theme}/
概要:
  - market-analysis.md: {市場規模のサマリー}
  - trend-report.md: {主要トレンド}
  - competitor-analysis.md: {競合数}社分析済み
  - sources.md: {ソース数}件
注意事項: {データの制約、古いデータの有無}
```

## 追加データ依頼テンプレート（Planner → Researcher）

```
依頼: {何を追加調査してほしいか}
理由: {なぜ必要か（企画書のどの部分で使うか）}
期待する成果物: {どんな形式で欲しいか}
参考: {検索キーワードのヒント等}
```

## 企画書事前相談テンプレート（Planner → Editor）

```
企画名: {名前}
方向性: {コンセプト概要}
差別化ポイント: {競合との違い}
懸念点: {自信がない部分、フィードバックが欲しい部分}
```

## レビューフィードバックテンプレート（Editor → Planner）

```
対象: {企画書名} ({ファイルパス})
評価: APPROVE / REQUEST_CHANGES
良い点: {具体的に}
改善点:
  1. {指摘 1}: {理由と改善案}
  2. {指摘 2}: {理由と改善案}
優先度: {どの改善を優先すべきか}
```
