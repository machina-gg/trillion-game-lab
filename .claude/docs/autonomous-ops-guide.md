# 自律運営ルール 詳細ガイド

社訓を実践するための具体的な運用ルール。

## 権限不足時の対応

エージェントが権限不足でブロックされた場合：

1. PM に報告する
2. 作業をブロックせず、次のタスクに移る
3. PM またはオーナーが申請を確認・対応する

## フィードバック受領時の対応

人からの指摘・フィードバックを受けた場合：

1. PM が即座に記録する
2. 反映先（CLAUDE.md、skills、docs 等）を明記する
3. **フィードバックを分類する**（下記参照）
4. 分類に応じて適切なタイミングで反映する

## フィードバック分類と反映フロー

フィードバックは受領時に以下の2種類に分類する。分類を誤ると、不要なルールが増えてノイズになるか、重要な学びが失われる。

### 分類基準

| 種類 | 判断基準 | 反映先 | 反映タイミング |
|------|---------|--------|--------------|
| **継続ルール** | 今後も繰り返し意識すべきこと。同じミスが再発しうる場合。 | CLAUDE.md / skills / checklists | PM が**即時反映** |
| **一回限り対応** | 今回の作業で対応すれば完了するもの。構造的な問題ではない場合。 | 該当ファイルを直接修正 | PM が**即時対応**して reflected に更新 |

### 判断の具体例

**継続ルールにすべきもの:**
- 「リサーチにはソースURLを必ず付けて」→ リサーチ時に毎回意識 → チェックリストに追加
- 「競合は3社以上分析して」→ 継続的な基準 → research-quality チェックリストで強調
- 「企画書にビジネスモデルを書いて」→ 常に意識 → proposal テンプレートに追加

**一回限り対応で済むもの:**
- 「このリサーチのデータが古い」→ 更新すれば完了
- 「この企画書の差別化が弱い」→ 修正すれば完了
- 「この機能を MVP に追加して」→ 追加すれば完了

### 反映フロー

```
フィードバック受領
  ↓
1. PM が記録（即座に）
  ↓
2. 分類判断: 継続ルール or 一回限り？
  ↓
┌─ 継続ルール ──────────────────────┐
│ PM が CLAUDE.md / skills / checklists│
│ に反映                              │
└──────────────────────────────────┘
┌─ 一回限り対応 ────────────────────┐
│ PM が該当作業を実施（または       │
│ エージェントに委譲）               │
│ （ルール化は不要）                  │
└──────────────────────────────────┘
  ↓
3. サイクル完了時: 未反映アイテムの最終確認（漏れ防止）
```

### 注意事項

- **迷ったら継続ルール寄りで判断する**: ルールは後から削除できるが、失われた学びは取り戻せない
- **一回限り対応でも、根本原因がプロセスにある場合は継続ルール化する**: 例えば「リサーチが古い」は一回限りだが、「データ更新フロー」は継続ルール

## 振り返りプロセス

サイクル完了後、PM は以下の振り返りを実施する：

### 手順

1. **Good / Bad の洗い出し**: うまくいったこと・問題を列挙
2. **根本原因の分析**: Bad の原因を特定
3. **改善策の立案**: 具体的なアクション + 反映先を決定
4. **即時反映**: CLAUDE.md や settings.json に改善を反映
5. **反映確認**: 変更が正しく適用されたことを確認

### 反映先の判断基準

| 改善の種類 | 反映先 | 実行者 |
|-----------|--------|--------|
| ワークフロー・ルール変更 | `CLAUDE.md` | PM |
| ツール権限・環境設定 | `.claude/settings.json` | PM |
| リサーチ品質基準 | `.claude/checklists/research-quality.md` | PM |
| 企画書品質基準 | `.claude/checklists/proposal-quality.md` | PM |
| CI / GitHub 設定の変更 | `.github/` | PM |
