# PR 作成チェックリスト

PR を作成する前に確認すること。

## コンテンツ品質

- [ ] ドキュメントの内容が正確で、誤字脱字がない
- [ ] ソースURLが有効で正しいリンクになっている
- [ ] データが最新で、出典が明記されている
- [ ] テンプレートに従ったフォーマットで書かれている

## Git

- [ ] コミットメッセージが規約に従っている（feat/fix/docs/style/refactor/chore）
- [ ] 不要なファイル（`.env`, `.DS_Store`）がコミットされていない
- [ ] feature ブランチが最新の develop から分岐している
- [ ] マージコンフリクトが解消されている

## Issue 紐付け

- [ ] **Issue が先に作成されている**（PR より Issue が先）
- [ ] PR 本文に `Closes #N` で Issue が紐付けられている
- [ ] 1 Issue = 1 PR になっている（PR が大きすぎないか確認）

## PR 本文

- [ ] タイトルが 70 文字以内
- [ ] Summary に変更の概要がある
- [ ] 関連 Issue へのリンクがある

## ファイル所有権

- [ ] 変更ファイルが自分の担当範囲内か確認した（CLAUDE.md「ファイル所有権」参照）
- [ ] PM 管轄ファイルと research/proposals/ が同一 PR に混在していない
- [ ] 混在がやむを得ない場合は `override:ownership` ラベルを付与した

## レビュー（PR 作成後に PM が実行）

- [ ] PM が変更内容に応じたレビュアーを特定した（リサーチ/企画→Editor、運営→PM）
- [ ] レビュアーエージェントを起動して `/review-pr` を実行した
- [ ] レビュアーが `gh pr review` でコメントを投稿した
- [ ] REQUEST_CHANGES の場合、修正を完了して再レビューを受けた
- [ ] APPROVE を受けてからマージした
- [ ] **CI が全パスしていることを `gh pr checks` で確認した（CI 未通過マージは厳禁）**
